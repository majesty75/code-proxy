# SYSTEM — Software Requirements Specification

## 1. Introduction

### 1.1 Purpose
This SRS defines the requirements for the UTA Test-Log Analytics & Visualization System — a distributed platform that ingests, parses, stores, and visualizes test logs from Samsung UTA (Unified Test Automation) servers in real-time, with historical analysis and future AI/ML capabilities.

### 1.2 Scope
The system covers the full lifecycle of test log data: collection from edge servers → real-time streaming → parsing → analytical storage → visualization → data science.

### 1.3 Definitions

| Term | Definition |
|------|-----------|
| UTA Server | Linux machine running Samsung's Unified Test Automation framework |
| Board / DUT | Device Under Test — a UFS/eMMC storage chip mounted in a rack slot |
| Slot | Physical location identifier: Rack-Shelf-Slot (e.g., R7S4-12) |
| Log File | Unstructured text file generated by a test execution on a board |
| Parser | Plugin module that extracts structured data from raw log lines |
| Hot Storage | Recent data (configurable, default 30 days) on fast disks, queried frequently |
| Cold Storage | Historical data (30d–2y) on slower disks, queried occasionally |

### 1.4 Assumptions
- UTA servers run Linux and have network connectivity to the main server.
- Main server runs Windows with WSL2 + Docker (or native Docker on Linux VMs).
- Log files are plain text, newline-delimited, with no guaranteed structure.
- Log filenames follow the convention documented in [log-naming.md](../log-naming.md).
- No authentication/RBAC is required in the initial release.

## 2. Functional Requirements

### 2.1 Log Collection (FR-COL)

| ID | Requirement | Priority |
|----|------------|----------|
| FR-COL-01 | System SHALL tail active log files in real-time from `/uta/UTA_FULL_Logs/` on each UTA server | P0 |
| FR-COL-02 | System SHALL detect new log files created in the watched directory and begin tailing automatically | P0 |
| FR-COL-03 | System SHALL enrich each log line with: server IP, log filename, line offset | P0 |
| FR-COL-04 | System SHALL resume from last read position after agent restart (checkpoint) | P0 |
| FR-COL-05 | System SHALL handle log rotation (file moved to backup directory) without data loss | P1 |
| FR-COL-06 | System SHALL auto-discover new UTA servers and deploy collection agents | P2 |

### 2.2 Message Transport (FR-MSG)

| ID | Requirement | Priority |
|----|------------|----------|
| FR-MSG-01 | System SHALL use a durable message broker between collection and processing | P0 |
| FR-MSG-02 | Messages SHALL be partitioned by `server_ip` for ordered processing per server | P0 |
| FR-MSG-03 | Broker SHALL retain unprocessed messages for at least 24 hours | P0 |
| FR-MSG-04 | System SHALL support at-least-once delivery semantics | P0 |
| FR-MSG-05 | System SHALL support exactly-once processing semantics (Flink checkpointing) | P1 |

### 2.3 Log Parsing (FR-PRS)

| ID | Requirement | Priority |
|----|------------|----------|
| FR-PRS-01 | System SHALL extract structured metadata from log filenames per [log-naming.md](../log-naming.md) | P0 |
| FR-PRS-02 | System SHALL support plug-and-play parser plugins (Python classes implementing a defined interface) | P0 |
| FR-PRS-03 | System SHALL route log lines to the appropriate parser based on filename/content matching | P0 |
| FR-PRS-04 | System SHALL gracefully handle unparseable lines (store raw line, mark as UNKNOWN) | P0 |
| FR-PRS-05 | System SHALL support adding new parsers without restarting the entire pipeline | P1 |
| FR-PRS-06 | System SHALL detect test PASS/FAIL/ERROR from log content | P1 |

### 2.4 Data Storage (FR-STO)

| ID | Requirement | Priority |
|----|------------|----------|
| FR-STO-01 | System SHALL store parsed log events in a columnar analytical database | P0 |
| FR-STO-02 | System SHALL store test session metadata (from filename) separately from log events | P0 |
| FR-STO-03 | System SHALL support flexible schema for parsed fields (Map type) | P0 |
| FR-STO-04 | System SHALL implement hot/cold storage tiering with configurable time boundaries | P1 |
| FR-STO-05 | System SHALL auto-expire data beyond retention period (TTL) | P1 |
| FR-STO-06 | System SHALL NOT store raw log files centrally | P0 |

### 2.5 Visualization (FR-VIZ)

| ID | Requirement | Priority |
|----|------------|----------|
| FR-VIZ-01 | System SHALL provide real-time dashboards showing live test status | P0 |
| FR-VIZ-02 | System SHALL allow filtering by: server, slot, platform, firmware, execution type | P0 |
| FR-VIZ-03 | System SHALL show log event rate, error rate, and severity distribution | P0 |
| FR-VIZ-04 | System SHALL provide historical BI dashboards for firmware version comparison | P1 |
| FR-VIZ-05 | System SHALL support drill-down from summary to individual test sessions | P1 |
| FR-VIZ-06 | System SHALL support cross-product comparison analytics | P1 |

### 2.6 AI/ML (FR-AIM) — Future Phase

| ID | Requirement | Priority |
|----|------------|----------|
| FR-AIM-01 | System SHALL provide a data interface for ML model training (ClickHouse → feature export) | P2 |
| FR-AIM-02 | System SHALL support anomaly detection on live test streams | P2 |
| FR-AIM-03 | System SHALL support failure prediction based on historical patterns | P2 |

## 3. Non-Functional Requirements

### 3.1 Performance

| ID | Requirement | Target |
|----|------------|--------|
| NFR-PERF-01 | End-to-end latency (log write → Grafana visible) | ≤ 10 seconds |
| NFR-PERF-02 | Sustained ingestion throughput | ≥ 150,000 lines/sec (10 servers × 15 boards × 1000 lines/sec) |
| NFR-PERF-03 | Dashboard query response time (last 1 hour) | ≤ 2 seconds |
| NFR-PERF-04 | Dashboard query response time (last 30 days) | ≤ 10 seconds |
| NFR-PERF-05 | Historical query response time (cold, last 1 year) | ≤ 60 seconds |

### 3.2 Reliability

| ID | Requirement | Target |
|----|------------|--------|
| NFR-REL-01 | No data loss under normal operation | 99.99% |
| NFR-REL-02 | System recovers from single-node failure without manual intervention | Yes |
| NFR-REL-03 | Pipeline resumes from checkpoint after restart | Within 60 seconds |

### 3.3 Scalability

| ID | Requirement | Target |
|----|------------|--------|
| NFR-SCL-01 | Support up to 10 UTA servers concurrently | Phase 1 |
| NFR-SCL-02 | Support up to 50 UTA servers | Phase 2 (horizontal scaling) |
| NFR-SCL-03 | Storage scales linearly with number of servers/boards | Via ClickHouse sharding |

### 3.4 Constraints

| ID | Constraint |
|----|-----------|
| CON-01 | All software must be open-source (no proprietary licenses) |
| CON-02 | All infrastructure is on-premises (no cloud services) |
| CON-03 | Main server runs Windows with WSL2 + Docker |
| CON-04 | UTA servers run Linux |
| CON-05 | 5-10 concurrent dashboard users |
| CON-06 | Raw log files are NOT stored centrally |

## 4. Data Requirements

### 4.1 Data Volume Estimates

| Metric | Value | Calculation |
|--------|-------|-------------|
| Servers | 10 | Given |
| Boards per server | 15 | 150 total / 10 servers |
| Lines per second per board | 1,000 | Given |
| Total lines/sec | 150,000 | 10 × 15 × 1000 |
| Avg line size | 200 bytes | Estimated |
| Raw ingestion rate | ~30 MB/s | 150K × 200B |
| Daily raw volume | ~2.6 TB | 30 MB/s × 86400 |
| After ClickHouse compression (~10:1) | ~260 GB/day | Columnar + LZ4 |
| 30-day hot storage | ~7.8 TB | 260 GB × 30 |
| 1-year total (hot + cold) | ~95 TB | 260 GB × 365 |

### 4.2 Data Retention

| Tier | Duration | Storage Type | Query Performance |
|------|----------|-------------|-------------------|
| Hot | 0–30 days | SSD / fast HDD | < 2s for dashboard queries |
| Cold | 30 days–2 years | HDD / NAS | < 60s for analytical queries |
| Archive | > 2 years | Delete (TTL) | N/A |
